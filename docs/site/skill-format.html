SKILL.md Format

Universal YAML-based specification format for AI skill lifecycle management. Compatible with Claude, OpenAI GPTs, Cursor, Kimi, OpenCode, and any markdown-reading AI platform.

📋 YAML Frontmatter

Every SKILL.md begins with a YAML block delimited by --- lines. The block is parsed by the skill framework; all other platforms receive it as readable text.

name string Unique skill identifier in kebab-case required
version string Semantic version, always quoted: "2.15.0" required
description string Plain-text summary — flat string for cross-platform compatibility required
description_i18n map Bilingual descriptions: {en: "…", zh: "…"} optional
license string SPDX identifier (MIT, Apache-2.0, …) optional
author map {name: string, email?: string} optional
created / updated string ISO 8601 date, always quoted: "2024-01-15" optional
type string manager | worker | orchestrator | utility optional
tags list Universal taxonomy for discovery and filtering optional
interface map {input, output, modes[]} — declared capabilities optional
extends map Framework extensions: evaluation thresholds, security standard, evolution triggers optional
Date fields must be quoted. Unquoted 2024-01-15 is parsed by YAML as a date object, not a string, and will cause type errors in strict parsers. Always write "2024-01-15".

✅ Canonical Frontmatter Example

Use this as the authoritative template. Every field is commented with its purpose.

SKILL.md — frontmatter YAML 
---
name: my-skill
version: "1.0.0"                    # always quote to force string

# description — flat string read by ALL platforms
description: "One-line summary of what this skill does."

# description_i18n — bilingual detail (ignored by unknown tools)
description_i18n:
  en: "Full English description with capabilities and constraints."
  zh: "完整的中文说明，包含能力与约束。"

license: MIT
author:
  name: Your Name
  email: you@example.com
created: "2024-01-15"               # quoted ISO 8601
updated: "2024-01-15"

type: worker                           # manager|worker|orchestrator|utility

tags:
  - lifecycle
  - evaluation

interface:
  input:  user-natural-language
  output: structured-skill
  modes:  [create, evaluate]

extends:
  evaluation:
    metrics:    [f1, mrr]
    thresholds: {f1: 0.90, mrr: 0.85}
  security:
    standard:        CWE
    scan-on-delivery: true
  evolution:
    triggers: [threshold, time, usage]
---

🔌 Platform Compatibility

SKILL.md is designed as a universal text format. Every platform reads the whole file as instructions; the YAML block is additional machine-readable metadata.

Platform Reads frontmatter Reads body Recommended usage
Claude / Claude Code as text ✓ full Place in CLAUDE.md or reference via @SKILL.md in system prompt
OpenAI GPTs as text ✓ full Paste file content into the GPT Instructions field; frontmatter acts as structured intro
Cursor as text ✓ full Reference in .cursorrules via @SKILL.md, or paste body into rules
Kimi as text ✓ full Upload as knowledge file or paste into system prompt; frontmatter is ignored by Kimi runtime
OpenCode as text ✓ full Place in project root alongside CLAUDE.md; both are auto-loaded
OpenClaw as text ✓ full Upload as an agent instruction file; body sections map naturally to agent capabilities
GitHub Pages / Jekyll ✓ parsed ✓ rendered Works natively; YAML frontmatter drives page metadata. Dates must be quoted strings.
skill CLI (this framework) ✓ full parse ✓ full Native format — all fields are validated and used for evaluation, certification, and evolution
Key insight: The flat description string is the cross-platform compatibility anchor. Platforms that cannot parse YAML still get a clear, readable summary as the first field in the block. The richer description_i18n is ignored silently by tools that don't understand it.

🏷️ Supported Modes

Declare active modes in interface.modes:

create
evaluate
restore
security
optimize

Trigger keywords for each mode are bilingual — see refs/triggers.md for the full pattern list.

🔧 extends — Framework Extensions

Tool-specific configuration for evaluation, security, and evolution. Unknown fields are safely ignored by external platforms.

evaluation

extends.evaluation
evaluation:
  metrics: [f1, mrr]
  thresholds:
    f1:  0.90   # ≥ 90 % F1
    mrr: 0.85   # ≥ 85 % MRR

security + evolution

extends.security / evolution
security:
  standard:        CWE
  scan-on-delivery: true

evolution:
  triggers:
    - threshold
    - time
    - usage

📄 Markdown Body Sections

After the closing ---, content is organised into numbered sections. The § prefix makes sections machine-scannable while remaining human-readable.

§1.1 Identity required Name, role, purpose, design patterns, core principles, red lines
§1.2 Framework required Architecture overview, mode routing diagram
§1.3 LOOP required Plan-Execute-Summarize phases and exit conditions
§1.4 Mode Router optional Decision tree and confidence routing rules
§2.0 Multi-LLM Deliberation optional LLM roles, consensus protocol, timeout spec
§3.0 Inversion Pattern optional Requirements elicitation methodology (blocking rule)
§4.0 Audit Trail optional Required fields and JSONL storage specification
§5.0 Security Red Lines optional CWE violation table and ABORT protocol steps
§6.0 Usage Examples optional Concrete input/output examples per mode
§7.0 Self-Evolution optional Evolution triggers, usage metrics, decision logic

✨ Minimal Skill Template

The smallest valid SKILL.md — works on every platform:

my-skill/SKILL.md Markdown + YAML 
---
name: my-skill
version: "1.0.0"
description: "Describe what your skill does in one sentence."
license: MIT
author:
  name: Your Name
created: "2024-01-15"
tags: [evaluation]
interface:
  input:  user-natural-language
  output: structured-skill
  modes:  [create, evaluate]
---

## §1.1 Identity

**Name**: my-skill
**Role**: Describe the agent role
**Purpose**: What problem does this skill solve?

## §1.2 Framework

Describe the architecture in 2-3 sentences.

## §1.3 LOOP

Describe the step-by-step execution loop.

---

**Triggers**: **CREATE** | **EVALUATE**
